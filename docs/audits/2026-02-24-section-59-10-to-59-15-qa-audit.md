## 6. Code quality and robustness

### 6.1 Null and type safety

- **bonus_calc:** `_to_float` used for all numeric inputs; `personnel or []` and `eligible_jobs or []` guard against None. `job.get(...)` with defaults avoids KeyError. Docstrings state “Treats None/missing as 0.”
- **filter_eligible_period_jobs:** Defensive `(job or {}).get("status")`; empty `period_jobs` returns `[]`.

### 6.2 Division and numeric edge cases

- **compute_job_base_splits:** Division by `len(seller_ids)` and `len(executor_ids)` occurs only when `tech_id in seller_ids` / `tech_id in executor_ids`, so denominators are ≥ 1. No division by zero.
- **apply_seller_penalties:** `per_seller = penalty_total / len(seller_ids)` only when `seller_ids` is non-empty and `penalty_total > 0`; otherwise returns a copy of `splits`. Safe.
- **compute_period_pot:** Sums over lists; no division. Safe.

### 6.3 Immutability and side effects

- **apply_* functions:** Return new dicts; do not mutate input `splits` or `job`/`personnel`. Callers can chain without surprise.
- **filter_eligible_period_jobs:** Returns a new list; does not mutate `period_jobs`.

### 6.4 Test coverage

- **test_bonus_calc.py:** Covers `compute_job_gp` (basic and parts-run deduction) and Schedule Saver (seller-only gets 60%; seller + executor 60/40). Run with `python3 -m unittest tests.test_bonus_calc` from `backend/`. All four tests pass.
- **Gap:** No unit tests for `compute_period_pot`, `apply_callback_voids`, `apply_estimation_accuracy`, or `apply_seller_penalties`. Acceptable for this audit as regression risk is low (unused in production path); optional improvement for future.

## 7. Recommendations and notes

### 7.1 Test discovery (minor) — **Implemented**

- **Observation:** `test_bonus_calc.py` uses `sys.path.insert(0, str(Path(__file__).resolve().parent.parent))` so that `from app.bonus_calc import ...` works when the test file is run directly or via `python -m unittest tests.test_bonus_calc` from `backend/`. The docstring also says “discover -s backend/tests”, which from project root would require `-s backend/tests` and may need PYTHONPATH or running from `backend/` for `app` to resolve.
- **Recommendation:** Document in README or a short testing section that bonus calc tests are run from `backend/`: `python3 -m unittest tests.test_bonus_calc -v`. Optional: add a one-line script in `scripts/` to run backend tests so CI/local use is consistent.
- **Done:** README now includes a "Backend unit tests" subsection (run from root: `./scripts/run-backend-tests.sh`; or from `backend/`: `python3 -m unittest discover -s tests -p "test_*.py" -v`). Added `scripts/run-backend-tests.sh` that runs tests from `backend/` with optional venv activation. Updated `backend/tests/test_bonus_calc.py` docstring to reference the script and commands.

### 7.2 Canonical wiring (59.18.2)

- **Observation:** New calculation engine is implemented but not connected to the dashboard. Dashboard still uses provisional team pot and provisional ledger.
- **Note:** This is intentional per plan. 59.18.2 will use `filter_eligible_period_jobs`, `compute_period_pot`, and the pipeline (base splits → callback voids → estimation accuracy → seller penalties) and replace provisional hero/ledger values. At that time, a focused QA pass on dashboard UX (labels, closed-period lock, mobile/desktop parity) is recommended.
