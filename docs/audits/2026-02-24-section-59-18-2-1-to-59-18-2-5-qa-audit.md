## 2. Desktop vs mobile production

### 2.1 Intent

Changes must affect **mobile only**; desktop production UI must remain unchanged.

### 2.2 Verification

- **CSS scoping**
  - All new layout/touch/safe-area rules are under `body[data-viewport-mode="mobile"]` and/or target elements that exist only inside `#bonusRaceBoardMobile`.
  - `.bonus-race-board-mobile` has `display: none` by default (line ~4187) and `display: grid` only when `body[data-viewport-mode="mobile"]` (line ~4427). Desktop never shows the race board.
  - `.bonus-hero-grid` is hidden only on mobile (`body[data-viewport-mode="mobile"] .bonus-hero-grid { display: none }`). Desktop continues to show the hero grid (Team Pot, Expected Payout, My GP).
  - No global overrides were added that would change desktop bonus view or other views.

- **Shared vs mobile-only rules**
  - `.bonus-racer-bar` and `.bonus-racer-bar--filled` are global, but the only DOM that uses them is the leaderboard list inside `#bonusRaceBoardMobile`, which is not visible on desktop. No desktop regression.
  - `@media (prefers-reduced-motion: reduce)` applies to `.bonus-pot-gauge-fill`, `.bonus-racer-bar`, `.bonus-racer` globally; these elements only appear in the bonus race block (mobile-only visible). No impact on desktop layout.

- **HTML/JS**
  - New `id="bonusRaceHeading"` and `aria-labelledby="bonusRaceHeading"` on the leaderboard `<ol>` live inside the race section (hidden on desktop). No duplicate IDs; no desktop behaviour change.
  - JS changes (aria-label, title attributes, slice-bar class application) affect only content rendered for the bonus dashboard; when on desktop, the hero grid is shown and the race block is hidden, so no desktop regression.

**Finding:** **PASS.** Desktop production is unaffected.

### 2.3 Minor gap (non-blocking)

- **Safe-area top:** The race board section has `padding-left`, `padding-right`, and `padding-bottom` using `env(safe-area-inset-*)`. The bonus view container (`.technician-bonus-view-container`) uses `padding: 16px` with no `safe-area-inset-top`. On notched devices, the top of the bonus view (period banner / first card) could sit under the status bar. **Recommendation:** Consider adding `padding-top: max(16px, env(safe-area-inset-top))` to `.technician-bonus-view-container` for mobile (e.g. under a mobile-only media query or `body[data-viewport-mode="mobile"]`) in a follow-up. Not required for sign-off.

## 4. UI/UX best practices

### 4.1 Touch targets (Apple HIG)

- **Requirement:** Minimum ~44pt (44px) touch targets for primary interactive elements on mobile.
- **Implementation:** Mobile-only rules set:
  - `.bonus-race-board-mobile .bonus-tally-btn`: `min-height: 44px`, `min-width: 44px`, padding 10px 14px.
  - `.bonus-race-board-mobile .bonus-tally-value`: `min-height: 44px`, `min-width: 44px`, inline-flex with padding 8px.
  - `.bonus-race-leaderboard .bonus-badge-chip` and `.bonus-effects-summary .bonus-badge-chip`: `min-height: 44px`, padding 10px 14px.
- **Finding:** **PASS.** All primary tap targets in the mobile GP Race block meet or exceed 44px.

### 4.2 Safe-area insets

- **Implementation:** `body[data-viewport-mode="mobile"] .bonus-race-board-mobile` has:
  - `padding-left: env(safe-area-inset-left, 0)`
  - `padding-right: env(safe-area-inset-right, 0)`
  - `padding-bottom: max(12px, env(safe-area-inset-bottom, 0))`
- **Finding:** **PASS** for left, right, and bottom. Top noted in §2.3 as optional follow-up.

### 4.3 Reduce motion

- **Implementation:**
  - `body.a11y-reduce-motion` continues to disable transitions on `.bonus-racer`, `.bonus-racer-bar`, `.bonus-pot-gauge-fill`.
  - New: `@media (prefers-reduced-motion: reduce)` disables transitions on the same elements so system preference is respected even when the app a11y class is not set.
  - `setBonusCurrencyValue` already used `shouldReduceBonusMotion()` to skip the count-up animation when reduce-motion is preferred.
- **Finding:** **PASS.** Visual motion (gauge, bar, racer, currency) is disabled when user prefers reduced motion.

### 4.4 Accessibility (ARIA, focus, semantics)

- **Leaderboard:** `<ol id="bonusRaceLeaderboard" aria-labelledby="bonusRaceHeading">` and `<h2 id="bonusRaceHeading">The Race</h2>` give the list an accessible name.
- **Buttons:** Leaderboard "View details" has `aria-label="View details"`. Tally and tally-value elements already had appropriate `aria-label`/`role`.
- **Tooltips:** All bonus chip buttons (effects summary, ledger role/sniper/penalty, leaderboard) now have `title` set to the same text as `data-bonus-tooltip`, improving native tooltip and assistive tech support.
- **Announcer:** `#bonusRaceAnnouncer` has `aria-live="polite"` and `aria-atomic="true"` (existing).
- **Finding:** **PASS.** ARIA and tooltip usage align with best practices.

### 4.5 Slice-bar animation (59.18.2.3)

- **Implementation:** Leaderboard bars use `--bar-width` and start at `width: 0`; after render, `requestAnimationFrame` adds `.bonus-racer-bar--filled` so width animates to `var(--bar-width)`. When `prefers-reduced-motion: reduce` or `body.a11y-reduce-motion`, transitions are disabled so the bar updates instantly.
- **Finding:** **PASS.** Animation is optional and respects reduce-motion.

### 4.6 Badge/streak/penalty icon mapping and tooltips (59.18.2.4)

- **Implementation:** Icon mapping is consistent (🏅 Do It All, 🎯 Sniper, 🔥 Hot Streak, ⚠️ Flat Tire, 🚩 Red Flag, 🧾 Seller, 🛠 Executor, 🤝 Co-Seller/Co-Executor). All relevant buttons have `title` and `data-bonus-tooltip` with the same content; user-facing and tooltip text are escaped via `escapeHtml()`.
- **Finding:** **PASS.** Consistent mapping and safe tooltip content.

### 4.7 Security (XSS)

- **Verification:** All new dynamic content in `title` and `data-bonus-tooltip` attributes is passed through `escapeHtml()` (e.g. `title="${escapeHtml(chip.tooltip)}"`, `title="${escapeHtml(tooltipText)}"`).
- **Finding:** **PASS.** No new XSS surface identified.
